1. Introduction
1.1 Purpose
This Software Design Document (SDD) describes the architecture and detailed design of the
AI-Powered Insurance Simulator. It translates the functional and non-functional requirements
defined in the Software Requirements Specification (SRS) into a concrete technical design that
the development team can implement, test, and maintain.
The document presents the overall system architecture, the decomposition of the system into
components, the internal and external interfaces, and the detailed behaviour of the most critical
components. It is intended to be the bridge between the requirement (functional) view captured
in the SRS and the structural (implementation) view embodied in the source code repository.
Source code reference: github.com/minh-quang-pham-le/Insurance-Simulator. The design
described here reflects the implementation present in that repository.
1.2 Scope
The AI-Powered Insurance Simulator is a simplified fintech platform that teaches users the
concepts of risk management and micro-insurance. Users browse and purchase short-term
parametric policies (flight delay, crop/weather, gadget, natural-disaster and rainfall coverage)
using a virtual currency (SimCoin) held in a virtual wallet. An automated parametric claims
engine settles claims using real-world data triggers, an AI chatbot assists purchase decisions,
and a risk engine estimates event probabilities from historical data.
The platform serves two audiences through two separate single-page applications backed by
one REST API and one database: an end-user Web App and an administrator Dashboard. This
SDD covers the design of the backend services, both frontends, the data model, and the
integrations with external data providers. It does not cover real financial transactions; all
currency and payouts are simulated for educational purposes.
Intended audience: project stakeholders, the product owner (course instructor), and the
development team (designers, programmers and testers).

3. System Architectural Design
3.1 System Description
The system is a web-based platform composed of four cooperating runtime units: two
browser-based client applications, one backend API server, and one relational database. The
backend additionally integrates with three external services (a weather API, a flight-status API,
and an LLM provider) that supply the data and intelligence behind the parametric and AI
features.
At a high level, an end user interacts with the User Web App to browse products, run risk
simulations, purchase policies, and manage a virtual wallet. An administrator uses the Admin
Dashboard to manage products, review KYC, monitor claims, and oversee the parametric
engine. Both clients call the same REST API, which encapsulates all business logic and is the
only component that touches the database and the external providers.

3.2 System Architecture
The platform follows a layered client/server architecture with a clear separation between
presentation, application logic and data:
• Presentation tier: Two independent Vue 3 + Vite single-page applications (user and
admin). Each is organised into views, reusable components, Pinia stores for state, and a
thin service layer that wraps Axios HTTP calls to the API.
• Application tier: A FastAPI server organised in three layers — routers (HTTP endpoints
and request/response validation via Pydantic schemas), services (business logic), and
models (SQLAlchemy ORM entities). Cross-cutting concerns such as JWT authentication
are handled by middleware.
• Data tier: PostgreSQL 15, accessed exclusively through the SQLAlchemy ORM; schema
evolution is managed with Alembic migrations.
• Integration tier: Dedicated service modules call external providers (weather, flight status)
and the Gemini LLM, isolating third-party concerns from core logic and recording calls in
an API monitor log.
The architecture is deployed as containerised services orchestrated by Docker Compose. Each
of the four runtime units (database, backend, user frontend, admin frontend) runs in its own
container, which supports horizontal scalability of the stateless backend as required by the SRS.
Architectural rationale and alternatives: A layered monolithic API was chosen over a
microservices architecture because the project scope, team size and timeline favour simplicity, a
single shared data model, and straightforward transactional consistency (important for wallet
operations). The router/service/model split keeps the codebase modular enough that
components can later be extracted into separate services if needed. A purely server-rendered

approach was rejected in favour of SPAs to deliver the interactive simulation and dashboard
experiences described in the SRS
3.3 Design Constraints
3.3.1 General constraints
• The project must be completed within the Introduction to Software Engineering course
timeline by a small team, favouring well-known frameworks over custom infrastructure.
• All money, premiums and payouts are simulated (SimCoin); the system must carry no
real-world financial or legal obligation.
• Critical operations (policy purchase, wallet credit after a claim) must satisfy ACID
properties; failures must roll back to prevent inconsistent wallet states.
3.3.2 Hardware constraints
• The system targets commodity server/container hosts; no specialised hardware (GPU,
etc.) is required. ML risk models are lightweight and run on CPU.
• Clients are standard desktop/laptop web browsers; no native mobile build is in scope.
3.3.3 Software constraints
• Backend: Python 3.10+, FastAPI, SQLAlchemy 2.0, Pydantic, Alembic. Database:
PostgreSQL 15.
• Frontend: Vue 3, Vite, TailwindCSS, PrimeVue, ApexCharts, Pinia, Axios.
• AI / external services: Google Gemini 2.0 Flash (chatbot), OpenWeatherMap and
AviationStack (trigger data); scikit-learn / joblib for the risk models.

The API returns HTTP status codes consistently: 200 for successful reads, 201 for created

6. Detailed Design
6.1 Introduction
This section describes the internal design of the principal backend and frontend-facing
components. The design follows the Router-Service-Model decomposition established in
Section 4. Routers own HTTP contracts, services own business rules and transaction
orchestration, and models own persistence structure.
The SRS emphasizes three critical qualities: secure access control, data consistency for
virtual-financial operations, and rapid automated claim settlement. Therefore the most detailed
design is provided for authentication, wallet, policy purchase, risk/simulation, claims, external
triggers, chatbot, notifications, and administration.
